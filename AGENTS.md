# Project Instructions

- The active GitHub checkout for this project is `active-github-repo/mitochondrial-deletion-discovery`. Use that subfolder for repository edits unless the user explicitly asks to work in the outer scratch tree or another copy.
- Before giving the user any Snakemake command to run, perform a dry-run of that exact command first, or explicitly state why a dry-run is impossible.
- For reruns from existing intermediate data, verify the dry-run job list does not include unintended download or trimming rules before recommending the command.
- After changes that affect plotting, report generation, matrices, normalization, grouping, labels, or table columns, test the code and inspect the generated plots/report before calling the work done. Check that plots use biological columns rather than workflow metadata, legends and labels are not clipped or misleading, empty panels explain the real reason they are unavailable, group colors/orders are consistent, and report text matches the current workflow behavior.
- Keep workflow behavior general and configuration-driven. Do not hard-code dataset-specific sample names, accessions, or biological conclusions into rules or scripts.
- The main workflow report is remap-focused. The first pass is read-selection/provenance unless a user explicitly asks for separate first-pass split-read result reporting.
- The default first-pass selection mode is `whole_genome_mt_best`: competitively map against the full genome including mtDNA and pass reads with mitochondrial best/selected evidence into mitochondrial remapping. Keep `nuclear_unmapped_reads` and legacy `mt_evidence_reads` modes available and configuration-driven for sensitivity checks and reproducibility.
- Describe mitochondrial circular-remap results as coordinate-focused, alignment-directed deletion evidence with circular coordinate normalization. Do not imply remap calls are biological deletions without filtering, deduplication, annotation, and context.
- Treat minimap2 split/supplementary alignments as evidence that may support an inferred deletion model. Do not assume every split alignment is a biological deletion without directed circular coordinate normalization, filtering, deduplication, annotation, and context.
- Preserve the distinction between exact deletions and affected-feature categories. Exact deletions are coordinate-level events; affected-feature categories are deterministic labels derived from the deleted interval overlapping reference features in genomic order.
- Circular mtDNA handling is core correctness logic. Query order and strand define a directed retained adjacency, and reciprocal directions are distinct deletion models; never choose an arc because it is shorter. Changes touching coordinate conversion, reciprocal directions, origin-spanning intervals, rotated-reference deduplication, ambiguity handling, or deletion IDs need targeted tests.
- RNA-specific transcript-processing artifacts matter for RNA datasets. Keep expected transcript-junction filtering and annotation configurable; do not hard-code species- or dataset-specific processing rules into shared code, and do not describe the whole workflow as RNA-only.
- Canonical deletion objects should carry at least: exact deletion ID, left/right breakpoints, deleted size, wraps-origin flag, deleted interval, read support, normalized support, affected features, fully removed features, partial overlaps, flanking/nearest features, size class, known-deletion match, sample, and group metadata.
- For minimap2 remapping, put index-sensitive settings such as `-k` and `-w` in `mt_realign.minimap2_index_extra`, not only in `mt_realign.minimap2_extra`; otherwise a prebuilt `.mmi` can silently keep the old seed parameters.
- Metadata resolution should prefer local cached metadata when available. Do not require live NCBI access for reruns from existing intermediate data.
- Sensitivity analysis should use existing full-run mitochondrial-evidence reads and full-run deletion-supporting reads, not raw FASTQ prefix subsets, when the goal is to test deletion-detection settings for rare events.
- Configured sequence searches are supplementary literal-read checks for named deletion motifs. Keep them config-driven and do not hard-code human deletion sequences into scripts or rules.
- Support offline operation by allowing local reference/sample files in configuration while preserving URL/download support.
